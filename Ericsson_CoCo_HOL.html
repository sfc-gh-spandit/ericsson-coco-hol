AGENTS.md is a project briefing file that Cortex Code reads automatically at the start of every conversation. It grounds every prompt in your schema, table relationships, and data quality notes.

Prompt
Create an AGENTS.md file with the following content:

# SEC Insider Trading Analytics

## Goal
Build a Bronze → Silver → Gold data pipeline and a Cortex Agent from SEC insider trading data.
Database: SEC_DEMO | Schemas: BRONZE, SILVER, GOLD, APP

## Source Tables (SNOWFLAKE_PUBLIC_DATA_FREE.PUBLIC_DATA_FREE)
- SEC_INSIDER_TRADING_SECURITIES_INDEX → BRONZE.INSIDER_TRADES (~16.7M trades)
- SEC_INSIDER_TRADING_REPORTING_OWNERS_INDEX → BRONZE.INSIDER_OWNERS (~5.5M owners)
- SEC_CIK_INDEX → BRONZE.COMPANIES (~973K companies)
- COMPANY_INDEX → BRONZE.COMPANY_MASTER (~2.9M records)
- COMPANY_SECURITY_RELATIONSHIPS → BRONZE.SECURITIES (~969K)
- GEOGRAPHY_INDEX → BRONZE.GEOGRAPHY (~573K locations)

## Verified Relationships
- INSIDER_TRADES.ADSH = INSIDER_OWNERS.ADSH (1:1 per filing)
- INSIDER_TRADES.ISSUER_CIK = COMPANIES.CIK (99.7% match)
- COMPANIES.COMPANY_ID = COMPANY_MASTER.COMPANY_ID (71% — individuals excluded)
- COMPANIES.STATE_GEO_ID = GEOGRAPHY.GEO_ID (100% match)
- SECURITIES.COMPANY_ID = COMPANY_MASTER.COMPANY_ID (100% match)

## Known Data Quality Issues
- REPORTING_OWNER_RELATIONSHIP is an ARRAY column — flatten to comma-separated string for readability
- REPORTING_OWNER_ADDRESS is a VARIANT column containing PII — do not expose in analytics
- COMPANY_MASTER has ARRAY columns (LEI, GLOBAL_TICKERS) — handle or exclude
- ~35% null TRANSACTION_PRICE_PER_SHARE — expected for options exercises (no cash price)
- ~30% null TRANSACTION_DATE — exclude these from Silver (incomplete filings)
- Duplicate rows on (ADSH, _SECURITY_SEQUENCE) — the true unique key for trades is (ADSH, _SECURITY_SEQUENCE, REPORTING_OWNER_CIK, SECURITY_TITLE). Deduplicate on this composite key keeping most recent REPORTED_PERIOD_DATE

## Architecture
- **BRONZE**: Raw immutable copies of marketplace tables (CTAS, no transformations)
- **SILVER**: Cleaned, normalized dimension and fact tables. Use Dynamic Tables with TARGET_LAG = '1 hour', WAREHOUSE = COMPUTE_WH. Add PK/FK constraints after creation (Dynamic Tables don't support inline constraints).
- **GOLD**: Single denormalized wide table joining Silver fact with dimensions — optimized for analytics, no joins needed at query time
- **APP**: Semantic View (defines business dimensions and metrics over Gold) + Cortex Agent (natural language Q&A)

Bronze Layer

~15 min — Immutable source of truth

Medallion Architecture organizes data by quality: Bronze (raw, immutable) → Silver (cleaned, normalized) → Gold (analytics-ready). Bronze never transforms data — it's a 1:1 copy that serves as your audit trail and replay source.

CREATE TABLE AS SELECT (CTAS) creates an owned copy of marketplace data. We need owned tables because governance features (DMFs, tags, masking policies) only attach to tables you own.

Data Quality & Streams

~18 min — Automated monitoring + real-time simulation

Data Metric Functions (DMFs) are automated quality checks that attach directly to tables. Set a DATA_METRIC_SCHEDULE (e.g., '5 MINUTE') and Snowflake runs them on that cadence — no external orchestrator needed. Think of them as built-in data observability.

Streams provide native Change Data Capture (CDC). They track inserts, updates, and deletes on a table — no Kafka, no Debezium, no external connector needed.

Stored Procedures are warehouse compute units that execute SQL and scripting logic inside Snowflake. Combined with Streams, they enable event-driven pipelines entirely within the platform.

Silver Layer

~25 min — 3NF normalized via Dynamic Tables

Dynamic Tables are Snowflake's declarative transformation layer. You define WHAT you want (a SQL query), and Snowflake handles WHEN and HOW to refresh it. Set a TARGET_LAG (freshness SLA) and Snowflake schedules incremental refreshes automatically — no Airflow, no dbt required.

TARGET_LAG = '1 hour' means the data will never be more than 1 hour behind the source. Snowflake calculates the optimal refresh schedule and performs incremental updates where possible.

3NF (Third Normal Form) means each table has a clear primary key and no redundant data. This reduces storage, prevents update anomalies, and creates clean dimension/fact separation for the Gold layer.

Gold Layer

~10 min — Denormalized for analytics

Denormalized wide table — we flatten the Silver fact and dimension tables into a single pre-joined table. This eliminates all JOINs for every analytics query, which is critical for BI tools and Cortex Analyst that perform best against flat structures. (A Star Schema by contrast still uses separate fact/dimension tables with FK joins — we go one step further here by fully denormalizing everything into one table.)

Dynamic Table on Dynamic Table creates a DAG (Directed Acyclic Graph). Snowflake automatically refreshes tables in dependency order: Bronze changes → Silver refreshes → Gold refreshes. No orchestrator needed.

5

AI Layer

~15 min — From SQL to conversation

Semantic Views add a business-context layer on top of your tables. They define which columns are dimensions (things you filter/group by) and which are metrics (things you measure). Without this, AI has no way to know that TRANSACTION_VALUE should be summed, or that SIC_CODE_CATEGORY means industry.

Cortex Analyst converts natural language questions into accurate SQL using the Semantic View definitions. It can only query columns you've explicitly defined — no hallucination of non-existent fields.

Cortex Agents orchestrate tools to answer user questions. One of those tools is cortex_analyst_text_to_sql, which uses the Semantic View to convert natural language into SQL. Users chat with the agent in Snowflake Intelligence; the agent calls Cortex Analyst under the hood and returns results conversationally.

1

Create Semantic View

Prompt
Create a semantic view SEC_DEMO.APP.INSIDER_TRADING_ANALYTICS over our Gold table. Define dimensions for everything users would filter or group by (dates, transaction types, company info, industry, geography, owner details) and metrics for what they'd measure (trade values, share volumes, averages, counts).

What's happening: Cortex Code inspects the Gold table schema and decides which columns are dimensions vs. metrics based on your description. It writes the CREATE SEMANTIC VIEW statement with appropriate COMMENTs — these comments help the AI understand the business meaning of each field.

2

Create Cortex Agent

Prompt
Create a Cortex Agent called SEC_DEMO.APP.INSIDER_TRADING_AGENT that uses our semantic view for natural language analytics over the insider trading data.

What's happening: The agent is configured with cortex_analyst_text_to_sql as its tool, pointed at your Semantic View. When users ask questions, the agent calls Cortex Analyst to generate SQL, executes it, and returns results conversationally.
